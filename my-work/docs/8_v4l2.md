# 8. 在 Buildroot QEMU 中搭建、模拟和观测 V4L2 软件栈

本文记录一次完整实验：在 Buildroot 2023.11.1 生成的 ARM64 QEMU 虚拟机中启用 V4L2/media 内核栈，加载 `vivid` 和 `vimc` 虚拟摄像头/虚拟 media pipeline 驱动，并用 `ftrace`、`function_graph`、`kprobe` 观测 V4L2/videobuf2 调用路径。

本文中的命令和输出来自当前机器的真实操作：

```text
Buildroot: /home/luckfox/workspace/buildroot-2023.11.1
实验目录:  /home/luckfox/workspace/buildroot-study/my-work
内核版本:  Linux 6.1.44
目标架构:  aarch64
QEMU 机型: qemu-system-aarch64 -M virt -cpu cortex-a53
```

## 1. 结论

Buildroot 编译出的 QEMU 虚拟机可以模拟和观测 V4L2 子系统的软件路径：

```text
userspace
  v4l2-ctl / yavta / ffmpeg
    |
    | ioctl(VIDIOC_REQBUFS / QBUF / STREAMON / DQBUF / STREAMOFF)
    v
kernel VFS/ioctl
    |
    v
V4L2 core
  v4l2_ioctl
  video_ioctl2
  video_usercopy
  __video_do_ioctl
    |
    v
videobuf2
  vb2_ioctl_reqbufs
  vb2_ioctl_qbuf
  vb2_ioctl_streamon
  vb2_ioctl_dqbuf
  vb2_ioctl_streamoff
  vb2_core_*
    |
    v
virtual driver
  vivid / vimc
```

但它不是在模拟某块真实 SoC 的完整硬件链路。比如下面这种真实摄像头链路，QEMU `virt` 机默认没有对应硬件模型：

```text
camera sensor -> MIPI CSI-2 -> ISP -> DMA -> interrupt -> V4L2
```

因此本文验证的是 Linux V4L2/media/videobuf2 软件栈和虚拟驱动路径，不验证真实 MIPI、ISP、DMA、寄存器时序和硬件中断行为。

## 2. Kernel Fragment 如何生效

Buildroot 侧控制内核 fragment 的配置项是：

```text
BR2_LINUX_KERNEL_CONFIG_FRAGMENT_FILES
```

当前 Buildroot `.config` 中已经接入两个 fragment：

```bash
rg -n "BR2_LINUX_KERNEL_CUSTOM_CONFIG_FILE|BR2_LINUX_KERNEL_CONFIG_FRAGMENT_FILES" \
  /home/luckfox/workspace/buildroot-2023.11.1/.config
```

真实输出：

```text
534:BR2_LINUX_KERNEL_CUSTOM_CONFIG_FILE="board/qemu/aarch64-virt/linux.config"
535:BR2_LINUX_KERNEL_CONFIG_FRAGMENT_FILES="board/qemu/aarch64-virt/observe-kernel.config board/qemu/aarch64-virt/v4l2-kernel.config"
```

规则如下：

1. `BR2_LINUX_KERNEL_CUSTOM_CONFIG_FILE` 是基础内核配置。
2. `BR2_LINUX_KERNEL_CONFIG_FRAGMENT_FILES` 是追加的配置片段列表。
3. Buildroot 在内核配置阶段会按顺序合并：base config -> fragment 1 -> fragment 2 -> `olddefconfig`。
4. 如果这个变量已经写在 Buildroot `.config` 里，普通 `make` 在需要配置 Linux 包时会自动应用。
5. 如果 Linux 包已经配置/编译过，而你后来修改了 fragment 或 fragment 列表，建议显式运行：

```bash
make linux-reconfigure
make
```

这样可以强制重新合并内核配置、重新编译内核/模块，并重新生成 rootfs 镜像。

注意：`make linux-menuconfig` 打开的是合并之后的内核配置界面。它不会自动把你的交互式修改反写回 fragment。如果需要长期可复现，应把变更沉淀到 fragment 文件里。

## 4. Fragment 选择脚本

已新增脚本：

```text
/home/luckfox/workspace/buildroot-study/my-work/scripts/apply_kernel_fragments.sh
```

它用于选择性更新 Buildroot `.config` 中的 `BR2_LINUX_KERNEL_CONFIG_FRAGMENT_FILES`。
脚本会根据自身路径定位默认 Buildroot 目录，因此可以用绝对路径从任意工作目录单独运行；也可以用 `--buildroot DIR` 显式指定 Buildroot 树。

列出可用 fragment：

```bash
cd /home/luckfox/workspace/buildroot-study/my-work
./scripts/apply_kernel_fragments.sh --list
```

真实输出：

```text
[fragments] Buildroot: /home/luckfox/workspace/buildroot-2023.11.1
[fragments] Fragment dir: board/qemu/aarch64-virt
  board/qemu/aarch64-virt/observe-kernel.config
  board/qemu/aarch64-virt/v4l2-kernel.config
```

查看当前已应用 fragment：

```bash
./scripts/apply_kernel_fragments.sh --show
```

真实输出：

```text
[fragments] Current BR2_LINUX_KERNEL_CONFIG_FRAGMENT_FILES:
  board/qemu/aarch64-virt/observe-kernel.config
  board/qemu/aarch64-virt/v4l2-kernel.config
```

选择性应用：

```bash
./scripts/apply_kernel_fragments.sh --set observe,v4l2 --reconfigure
```

应用全部 fragment 并重新构建完整镜像：

```bash
./scripts/apply_kernel_fragments.sh --all --reconfigure --build
```

只修改 Buildroot `.config`，不触发内核重配：

```bash
./scripts/apply_kernel_fragments.sh --set v4l2 --no-sync
```

脚本支持的主要参数：

| 参数 | 作用 |
|------|------|
| `--list` | 列出 fragment 目录下的可用片段，自动排除基础 `linux.config`。 |
| `--show` | 显示当前 Buildroot `.config` 中的 fragment 列表。 |
| `--set a,b` | 用指定列表替换当前 fragment 列表。 |
| `--add a,b` | 在当前列表后追加 fragment。 |
| `--remove a,b` | 从当前列表中移除 fragment。 |
| `--all` | 应用目录中全部 fragment。 |
| `--clear` | 清空 fragment 列表。 |
| `--reconfigure` | 执行 `make linux-reconfigure`。 |
| `--build` | 执行完整 `make`。 |
| `--buildroot DIR` | 指定 Buildroot 工程路径。 |

## 12. 用 function_graph 观察 V4L2 ioctl 层级

命令：

```sh
cd /sys/kernel/tracing
echo 0 > tracing_on
echo nop > current_tracer
echo > trace
echo > set_ftrace_filter
echo > set_ftrace_notrace
echo > set_graph_function
echo function_graph > current_tracer
echo video_ioctl2 > set_graph_function
echo 1 > options/funcgraph-proc
echo 1 > tracing_on

v4l2-ctl -d /dev/video0 --stream-mmap --stream-count=2 --stream-to=/tmp/trace-fgraph.raw

echo 0 > tracing_on
grep -E 'video_ioctl2|__video_do_ioctl|v4l2_|vb2_|vivid_' trace | head -80
```

真实输出摘录：

```text
 0)  v4l2-ct-153   |               |  video_ioctl2() {
 0)  v4l2-ct-153   |               |      __video_do_ioctl() {
 0)  v4l2-ct-153   |               |  video_ioctl2() {
 0)  v4l2-ct-153   |               |      __video_do_ioctl() {
 0)  v4l2-ct-153   |               |          v4l2_query_ext_ctrl() {

 0)  v4l2-ct-153   |               |  video_ioctl2() {
```

解释：

`v4l2-ctl` 在正式 streaming 前会查询大量能力、control、selection，所以 trace 前半段出现很多 `v4l2_query_ext_ctrl()` 是正常现象。这个实验用于证明 ioctl 层级：

```text
v4l2_ioctl
  -> video_ioctl2
      -> video_usercopy
          -> __video_do_ioctl
              -> specific V4L2 operation
```

## 13. 用 kprobe 观察 vb2 特定函数和调用栈

`kprobe` 更适合观察特定函数。下面只盯住 videobuf2 的核心队列函数：

```sh
cd /sys/kernel/tracing
echo 0 > tracing_on
echo nop > current_tracer
echo > trace
for e in events/kprobes/*/enable; do [ -e "$e" ] && echo 0 > "$e"; done
echo > kprobe_events

echo 'p:probe_vb2_reqbufs vb2_core_reqbufs' >> kprobe_events
echo 'p:probe_vb2_qbuf vb2_core_qbuf' >> kprobe_events
echo 'p:probe_vb2_dqbuf vb2_core_dqbuf' >> kprobe_events
echo 'p:probe_vb2_streamon vb2_core_streamon' >> kprobe_events
echo 'p:probe_vb2_streamoff vb2_core_streamoff' >> kprobe_events

cat kprobe_events
for e in events/kprobes/probe_vb2_*/enable; do echo 1 > "$e"; done
echo stacktrace > events/kprobes/probe_vb2_qbuf/trigger
echo 1 > tracing_on

v4l2-ctl -d /dev/video0 --stream-mmap --stream-count=2 --stream-to=/tmp/trace-kprobe.raw

echo 0 > tracing_on
grep -E 'probe_vb2|<stack trace>|=>|video_ioctl2|vb2_|v4l2_|vivid_' trace | head -220
```

真实 `kprobe_events`：


这段是本文最关键的观测结果。它证明了 capture streaming 中的核心调用顺序：

```text
VIDIOC_REQBUFS  -> vb2_core_reqbufs
VIDIOC_QBUF     -> vb2_core_qbuf
VIDIOC_STREAMON -> vb2_core_streamon
VIDIOC_DQBUF    -> vb2_core_dqbuf

VIDIOC_DQBUF    -> vb2_core_dqbuf

```

其中 `vb2_core_qbuf` 的栈明确显示：

```text
userspace ioctl
  -> __arm64_sys_ioctl
  -> v4l2_ioctl
  -> video_ioctl2
  -> video_usercopy
  -> __video_do_ioctl
  -> v4l_qbuf
  -> vb2_ioctl_qbuf
  -> vb2_core_qbuf
```

清理 kprobe：

```sh
echo 0 > events/kprobes/probe_vb2_qbuf/enable
echo '!stacktrace' > events/kprobes/probe_vb2_qbuf/trigger
for e in events/kprobes/probe_vb2_*/enable; do echo 0 > "$e"; done
echo > kprobe_events
```

## 17. 学习路径建议

建议按这个顺序做实验：

1. 先用 `vivid` 跑通 `/dev/video0`、格式设置、mmap 抓帧。
2. 用 `kprobe` 观察 `vb2_core_reqbufs/qbuf/dqbuf/streamon/streamoff`。
3. 用 `function_graph` 看 `video_ioctl2 -> __video_do_ioctl` 的层级。
4. 再加载 `vimc`，用 `media-ctl -p` 学习 sensor/subdev/pad/link。
5. 最后再进入 GDB 或 eBPF/kprobe，做参数级、结构体级分析。

这套环境已经足够分析 V4L2 core、media controller、videobuf2 和虚拟 capture driver 的软件行为。
