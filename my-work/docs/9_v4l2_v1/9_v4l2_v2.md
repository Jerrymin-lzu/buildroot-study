# 9. Buildroot QEMU 中的 V4L2 与观测性实验

本文把 V4L2 学习路线落实成可以在当前 Buildroot + QEMU 虚拟机里执行的实验。文中的命令和输出来自本机实际运行，输出做了必要删减，但没有改写为“示意输出”。

和 v1 相比，本版重点不只是“跑通命令”，而是训练读输出的方法：看到一段
`strace`、ftrace、kprobe、tracepoint、perf、火焰图或 sched 日志时，能判断它在证明什么、不能证明什么，以及下一步应该换哪种观测工具补证据。

实验环境：

```text
Buildroot: /home/luckfox/workspace/buildroot-2023.11.1
实验目录:  /home/luckfox/workspace/my_work
目标系统:  Buildroot aarch64 guest
内核版本:  Linux 6.1.44
虚拟机:    qemu-system-aarch64 -M virt -cpu cortex-a53 -smp 1
虚拟摄像头: vivid
```

本文的学习目标：

1. 看到一次 V4L2 抓帧如何从用户态 ioctl 进入内核。
2. 把 `VIDIOC_QBUF`、`VIDIOC_DQBUF` 等 ioctl 映射到 V4L2 core 和 VB2。
3. 使用 ftrace、tracepoint、kprobe、perf、trace-cmd 观察时序、热点和调度。
4. 为后续分析 CAMSS/CAMX/ISP 代码建立方法，而不是只记命令。

### 本版读输出的方法

读这类系统观测输出时，先分清“稳定信息”和“运行现场信息”。

稳定信息通常包括函数名、事件名、模块名、ioctl 名、调用关系、buffer 状态变化和线程角色。比如 `video_ioctl2`、`vb2_core_qbuf`、`sched_switch`、`VIDIOC_DQBUF` 这些名字背后对应的是内核路径或 ABI，具有解释价值。

运行现场信息通常包括 PID、时间戳、地址、百分比、样本数和偶发 warning。它们很有用，但需要结合上下文。比如 `v4l2-ctl-206` 的 `206` 下次会变，`0x55610d11b0` 这种地址也会变；但“用户线程进入 `D/S`，随后 vivid capture 线程唤醒它”这个结构才是可迁移的规律。

每段输出按四层读：

1. 事实层：命令实际看到了什么对象？模块、设备、函数、事件、线程是否出现？
2. 路径层：这些对象是否能串成 `userspace -> ioctl -> V4L2 core -> VB2 -> driver`？
3. 时间/比例层：耗时、帧间隔、样本占比、调度切换是否和预期一致？
4. 证据边界：这段输出能证明什么，不能证明什么？需要 `strace`、tracepoint、perf 还是 sched 事件补证？

后文每个“实际输出”都尽量按这个思路解释。读者以后遇到类似输出，不必记住本机的 PID 或地址，而是先找关键角色、关键路径和关键状态变化。

这里的规律是用三个互相独立的信号确认抓帧成功：

- 格式查询从 `640x360` 变成 `640x480`，说明 `S_FMT` 生效。
- `<<<<<` 有 5 个 `<`，说明用户态循环拿到了 5 帧。
- 文件大小等于 `Size Image * frame_count`，说明写出的数据量和格式匹配。

以后看类似输出，不要只看命令返回 0。要把“格式字段、打印帧数、文件字节数”三者交叉验证。若 `<` 数量对但文件大小不对，要怀疑输出路径或 bytesused；若文件大小对但帧率异常，要继续看 `DQBUF` 阻塞和调度。

### 1.2 输出含义

一次 mmap streaming 抓帧的用户态生命周期是：

```text
VIDIOC_QUERYCAP       查询设备能力
VIDIOC_REQBUFS        申请 buffer，这里申请 4 个 mmap buffer
VIDIOC_QUERYBUF       查询每个 buffer 的 offset/length
mmap                  v4l2-ctl 内部把 buffer 映射到用户态，strace 这里只过滤 ioctl 所以没显示
VIDIOC_QBUF           把 buffer 入队，交给驱动/VB2
VIDIOC_STREAMON       启动采集流
VIDIOC_DQBUF          等待并取回完成的 buffer
VIDIOC_QBUF           处理完一帧后重新入队
VIDIOC_STREAMOFF      停止流
VIDIOC_REQBUFS count=0 释放 buffer
```

最关键的是 `DQBUF`：

- 第一帧 `DQBUF` 只花了 `0.000570s`，因为启动后很快有 buffer ready。
- 后两帧分别花了 `0.193456s` 和 `0.193755s`。
- `vivid` 当前帧率是 5 fps，也就是每帧约 `0.2s`，所以 `DQBUF` 慢不是 bug，而是在等待下一帧完成。

读 `strace -ttT -e trace=ioctl` 输出时，先拆一行：

```text
17:16:01.045134 ioctl(3, VIDIOC_DQBUF, {... index=1 ...}) = 0 <0.193456>
```

- `17:16:01.045134` 是用户态发起 syscall 的时间。
- `ioctl(3, ...)` 里的 `3` 是 fd，不是设备号；真正设备要结合程序打开的是 `/dev/video0`。
- `VIDIOC_DQBUF` 是 V4L2 ABI 操作名，比数字 ioctl 更有解释价值。
- `{... index=1 ...}` 是 ioctl 参数，`index` 能和后面的 tracepoint buffer index 对上。
- `= 0` 表示 syscall 成功返回；负值或 `errno` 才是错误路径。
- `<0.193456>` 是 syscall 在用户态看来阻塞了多久。

类似输出的核心规律是先看序列，再看耗时。序列回答“程序做了什么”，耗时回答“慢在哪里”。对 mmap streaming，健康的基本顺序通常是 `REQBUFS -> QUERYBUF -> QBUF -> STREAMON -> DQBUF/QBUF 循环 -> STREAMOFF -> REQBUFS count=0`。如果 `DQBUF` 慢，要继续判断是正常帧间隔、驱动没完成 buffer，还是调度唤醒延迟。

### 1.3 学到什么

用户态看到的“摄像头慢”，常常具体表现为 `VIDIOC_DQBUF` 阻塞。后续定位要继续问：

- 是驱动没有按时完成 buffer？
- 是真实硬件中断没来？
- 是线程被调度打断？
- 是用户态处理太慢，导致 buffer 没及时重新 `QBUF`？

这一节先把问题锚定到用户态 syscall 时序上。

这一节把用户态和内核入口对应起来：

## 3. 抓 VB2 缓冲区路径

VB2 是 V4L2 的通用 buffer queue 框架。真实 camera 驱动通常不会自己从零管理所有 buffer，而是接入 VB2。

读 VB2 函数名时可以按命名分层：

- `vb2_ioctl_*`：从 V4L2 ioctl ops 进来的适配层，连接 `v4l_qbuf` 这类 V4L2 core 函数。
- `vb2_*`：VB2 V4L2 层的通用接口，常在 `videobuf2_v4l2` 模块里。
- `vb2_core_*`：真正维护 queue 和 buffer 状态机的核心层。
- `vb2_vmalloc_*`、`vb2_dc_*`：内存后端，决定 buffer 如何映射、分配或 DMA。

这套命名规律很重要。以后在真实驱动里看到 `vb2_core_qbuf`，说明已经进入 VB2 状态机；看到 `buf_queue` 或 driver 回调，才说明 buffer 交给具体驱动。

### 3.3 输出含义

这段栈把 V4L2 core 和 VB2 串起来了：

```text
userspace ioctl(VIDIOC_QBUF)
  -> __arm64_sys_ioctl
      -> v4l2_ioctl
          -> video_ioctl2
              -> video_usercopy
                  -> __video_do_ioctl
                      -> v4l_qbuf
                          -> vb2_ioctl_qbuf
                              -> vb2_core_qbuf
```

一次 2 帧抓取的核心 VB2 顺序是：

```text
REQBUFS    -> vb2_core_reqbufs
QBUF x4    -> vb2_core_qbuf
STREAMON   -> vb2_core_streamon
DQBUF      -> vb2_core_dqbuf
QBUF       -> vb2_core_qbuf
DQBUF      -> vb2_core_dqbuf
QBUF       -> vb2_core_qbuf
STREAMOFF  -> vb2_core_streamoff
REQBUFS 0  -> vb2_core_reqbufs
```

这条顺序是 VB2 的状态机视角。它和 `strace` 的 ioctl 顺序一致，但比 `strace` 更靠近内核内部。方法论上，`strace` 告诉你用户态请求了什么，kprobe 告诉你这个请求是否真的进入了 VB2 core。两者互相印证时，路径判断才稳。

### 3.4 清理 kprobe

切换到 tracepoint 实验前要清理动态 kprobe，否则输出会混在一起。

虚拟机内执行：

```sh
cd /sys/kernel/tracing
echo 0 > tracing_on
for e in events/kprobes/*/enable; do [ -e "$e" ] && echo 0 > "$e"; done
[ -e events/kprobes/probe_vb2_qbuf/trigger ] && echo '!stacktrace' > events/kprobes/probe_vb2_qbuf/trigger 2>/dev/null || true
echo > kprobe_events
cat kprobe_events
```

实际输出为空，表示 kprobe 已清理。

### 3.5 学到什么

V4L2 core 负责把 ioctl 分发到具体 operation，VB2 负责 buffer 状态机。以后读 camera 驱动时，要重点找驱动如何填充：

- `vb2_queue`
- `queue_setup`
- `buf_queue`
- `start_streaming`
- `stop_streaming`
- buffer done 路径，例如 `vb2_buffer_done()`

```text
vivid-000-vid-c-178 [000] ..... 423.315857: vb2_buf_done: ... index = 1, bytesused = 614400
```

- `index` 是 buffer 槽位，不等于帧号。这里 0、1、2、3 会循环复用。
- `sequence` 是帧序号，正常抓帧时应递增。这里 `v4l2_dqbuf` 看到 0、1、2。

- `owned_by_drv` 表示有多少 buffer 已交给驱动持有。`STREAMON` 后从 1 增到 4，说明初始 4 个 buffer 都交给驱动。

### 4.3 输出含义

这段事件流比函数调用更接近真实业务：

```text
v4l2-ctl          vb2_qbuf / v4l2_qbuf      用户态把 4 个 buffer 入队
v4l2-ctl          vb2_buf_queue             STREAMON 后 buffer 交给驱动
vivid-000-vid-c   vb2_buf_done              vivid 内核线程生成一帧并完成 buffer
v4l2-ctl          vb2_dqbuf / v4l2_dqbuf    用户态 DQBUF 取走完成帧
v4l2-ctl          vb2_qbuf                  用户态把该 buffer 重新入队
```

`vivid-000-vid-c` 是 vivid 的视频采集线程。真实硬件平台上，对应的完成路径可能来自中断 handler、tasklet、workqueue 或 kthread，最终也会走到 `vb2_buffer_done()` 一类逻辑。

类似 tracepoint 输出的分析方法是把 buffer 当成“状态机对象”：

```text
用户态 QBUF
  -> VB2 queued 增加
  -> STREAMON 后 buf_queue 交给驱动
  -> 驱动/线程/中断 buf_done
  -> 用户态 DQBUF 取回
  -> 用户态再次 QBUF
```

如果真实设备丢帧或卡住，就沿着这条状态机找断点：有没有 QBUF？有没有交给驱动？有没有 buf_done？done 后 DQBUF 有没有及时发生？sequence 是否跳号？这比单纯看函数名更接近 camera pipeline 的业务问题。

### 4.4 学到什么

tracepoint 适合回答：

- 哪个 buffer index 入队？
- 哪个线程完成了 buffer？
- buffer done 和用户态 `DQBUF` 的时间差是多少？
- 帧序号是否连续？
- streamoff 时是否还有未完成 buffer 被 flush？

这是以后分析真实 camera pipeline 最重要的视角。

```text
_IO_fwrite -> _IO_file_write -> write -> __arm64_sys_write
-> vfs_write -> generic_perform_write -> shmem_write_begin
```

这说明 workload 变成了“抓到帧后写文件”。`/tmp/perf.raw` 大小是 `12288000`，等于 `614400 * 20`，正好对应 20 帧 YUYV 数据。因此文件 I/O 和 shmem 页分配/写入成为热点是合理的。

### 5.2 输出含义

这个 perf 结果说明：如果把每帧都写到 `/tmp/perf.raw`，热点主要变成文件写入路径：

```text
v4l2-ctl
  -> _IO_fwrite
      -> write
          -> __arm64_sys_write
              -> vfs_write
                  -> generic_perform_write
                      -> shmem_write_begin
                      -> __arch_copy_from_user
```

这不是坏结果，而是一个重要提醒：性能分析必须考虑观测对象本身。你以为自己在分析 camera，实际上可能在分析“把帧写到文件”的成本。

类似 perf 输出的第一条方法论：先确认 workload 是否被观测方式改变。`--stream-to=/tmp/perf.raw` 会引入文件写入、页缓存、内存拷贝；如果目标是分析 camera ioctl 或驱动等待，这个输出就是“被写文件主导的实验”。它仍有价值，因为它告诉你写 raw 文件本身会成为瓶颈，但不能拿它直接代表 camera pipeline 的 CPU 热点。

`QBUF` 和 `REQBUFS` 的路径在前面的 ftrace/kprobe 里已经确认；如果查看未截断的
perf report，也可能看到类似分支：

```text

v4l_reqbufs -> vb2_ioctl_reqbufs -> vb2_core_reqbufs -> __vb2_queue_free
```

终端输出本身仍有 `write` 成本，因为 `v4l2-ctl` 会打印 `<` 和 fps。真实分析时可以降低日志输出，或者让 workload 更接近目标场景。

不写 raw 文件版本的读法是做对照实验：移除大块帧数据写入后，`_IO_fwrite/write/vfs_write` 不再主导，`ioctl -> v4l2_ioctl -> vb2_core_dqbuf` 路径就浮出来了。但它仍有少量 `write/redirected_tty_write/uart_write`，因为 `v4l2-ctl` 还会向终端打印 `<` 和 fps。

这说明 perf 结论要成对读：

```text
写 raw：热点 = 帧数据输出路径 + 内存/页缓存成本
不写 raw：热点 = ioctl/VB2 路径 + 少量终端输出
```

类似场景中，如果 perf 看到的热点和预期业务不一致，先问：程序有没有打印大量日志？有没有写文件？有没有额外 memcpy？有没有调试开关改变路径？性能分析的第一步不是解释热点，而是确认热点是否属于你真正想分析的 workload。

### 5.5 学到什么

perf 的价值不是“列出函数名”，而是回答：

- 时间主要花在文件写入、用户态处理、ioctl、驱动，还是调度等待？
- 调用栈里有没有 `copy_to_user`、`copy_from_user`、内存分配、cache 操作？
- 观测方式是否改变了热点？

火焰图的读法：

- 横向宽度表示采样占比，不是调用先后顺序。
- 宽的栈表示热点。
- 如果 `_IO_fwrite/write/vfs_write` 很宽，说明你主要在分析写文件。
- 如果 `vb2_core_dqbuf` 或调度相关栈很宽，要进一步结合 tracepoint 和 sched 事件分析等待原因。

`perf script` 和火焰图之间的关系是：

```text
perf.data
  -> perf script      展开每个采样点的调用栈
  -> stackcollapse    把相同调用栈聚合成 "a;b;c count"
  -> flamegraph       把聚合栈画成宽度比例
```

所以读火焰图时不要按左右顺序理解时间先后。横向位置没有“先发生/后发生”的含义，横向宽度才代表采样占比。纵向代表调用深度，下面是调用者，上面是被调用者。看到一个宽块时，要向下追它的父调用链，判断它属于用户态处理、syscall、VFS、VB2 还是调度路径。

本次 `head -80` 截到的是 `execve` 早期样本，说明只看前几十行 `perf script` 很容易误判。方法论上，`perf script | head` 只用于确认格式，不用于判断热点；热点应看 `perf report` 或完整火焰图。

字段规律：

- `sched_waking`：准备唤醒某个任务。
- `sched_wakeup`：任务已经进入可运行状态。
- `sched_switch`：CPU 实际从一个任务切到另一个任务。
- `prev_state=R`：前一个任务仍可运行，只是被切走。
- `prev_state=S`：前一个任务进入可中断睡眠。
- `prev_state=D`：前一个任务进入不可中断睡眠，常见于等待 I/O、buffer、驱动条件等。
- `next_comm=...`：接下来真正拿到 CPU 的任务。

分析规律是：`wakeup` 不等于马上运行，只有看到后续 `sched_switch ... ==> next_comm=目标线程`，才说明它实际获得 CPU。真实系统里，如果 `buf_done` 已经发生、`sched_wakeup` 也发生，但目标线程很久以后才 `sched_switch` 上 CPU，就要怀疑调度延迟或优先级问题。

### 7.2 输出含义

关键线程：

- `v4l2-ctl-206`：用户态抓帧进程。
- `vivid-000-vid-c-207`：vivid 的 capture 线程。

关键现象：

- `v4l2-ctl` 进入 `S` 或 `D` 状态，说明它在等待。
- `vivid-000-vid-c` 运行并唤醒 `v4l2-ctl`。
- 5683.681 到 5683.881 之间约 0.2 秒，对应 5 fps 的帧间隔。

因此 `DQBUF` 慢的直接原因是：用户态睡眠等待下一帧，vivid 线程在帧完成后唤醒它。

这段 sched 输出和前面的 tracepoint 可以合起来读：

```text
tracepoint: vivid-000-vid-c 产生 vb2_buf_done
sched:     vivid-000-vid-c 唤醒 v4l2-ctl
strace:    v4l2-ctl 的 VIDIOC_DQBUF 返回
```

这就是跨工具拼证据的方法。单独看 `strace` 只能知道 `DQBUF` 等了约 0.2 秒；单独看 tracepoint 只能知道 buffer done 发生；单独看 sched 只能知道谁唤醒谁。三者合起来，才能说清楚“用户态等待下一帧，vivid capture 线程完成 buffer 并唤醒用户态”。

调度 trace 用来回答：

- 谁唤醒了用户线程？
- 唤醒后是否马上运行，还是被其他线程抢占？
- 真实系统中帧间 jitter 是来自硬件、驱动，还是调度？

### 8.1 建议一次性采集的文件

### 8.2 用本次实验回答四个问题

问题 1：一次抓帧经历哪些 ioctl？

从 `/tmp/ioctl.log` 的实际输出看：

```text

VIDIOC_G_INPUT
VIDIOC_ENUMINPUT
VIDIOC_REQBUFS
VIDIOC_QUERYBUF
VIDIOC_QBUF
VIDIOC_STREAMON
VIDIOC_DQBUF
VIDIOC_QBUF
VIDIOC_STREAMOFF
VIDIOC_REQBUFS count=0
```

问题 2：这些 ioctl 在内核里对应哪些 V4L2/VB2 路径？

从 ftrace 和 kprobe 的实际输出看：

```text
__arm64_sys_ioctl
  -> v4l2_ioctl
      -> video_ioctl2
          -> video_usercopy
              -> __video_do_ioctl
                  -> v4l_qbuf / v4l_dqbuf / v4l_reqbufs
                      -> vb2_ioctl_qbuf / vb2_ioctl_dqbuf / vb2_ioctl_reqbufs
                          -> vb2_core_qbuf / vb2_core_dqbuf / vb2_core_reqbufs
```

问题 3：哪一步耗时最长？

从 strace 的实际输出看：

```text
VIDIOC_DQBUF = 0 <0.193456>
VIDIOC_DQBUF = 0 <0.193755>
```

`DQBUF` 耗时最长，因为它在等待下一帧。当前 vivid 帧率是 5 fps，理论帧间隔约 0.2 秒，和观测结果一致。

问题 4：是否存在调度延迟或内存拷贝热点？

从 sched trace 看：

```text
v4l2-ctl 进入 S/D 状态等待
vivid-000-vid-c 唤醒 v4l2-ctl
帧间隔约 0.2 秒
```

从 perf 看：

```text
写 raw 文件版本热点主要在 write/vfs_write/generic_perform_write/shmem_write_begin
不写 raw 文件版本可以看到 ioctl -> v4l2_ioctl -> vb2_core_dqbuf/qbuf/reqbufs
```

所以在本次 QEMU + vivid 实验中，长等待主要是帧率导致的 `DQBUF` 等待；如果把帧写到文件，文件写入和内存拷贝会成为明显热点。

把这些输出合起来，可以得到一张证据矩阵：

```text
strace
  证明：用户态调用了哪些 ioctl，每个 ioctl 阻塞多久
  边界：不知道内核里谁完成 buffer，也不知道 CPU 热点

ftrace / kprobe
  证明：ioctl 进入了哪些 V4L2/VB2 函数，调用链是否符合预期
  边界：不直接告诉 buffer index、sequence，也不适合单独判断帧间隔

V4L2/VB2 tracepoint
  证明：buffer 的入队、交给驱动、完成、出队时序
  边界：不直接告诉 CPU 热点，也不完整解释调度延迟

perf report / flamegraph
  证明：CPU 样本主要落在哪些调用栈
  边界：采样看到的是运行中的 CPU 消耗，不等于所有等待时间

sched trace / trace-cmd
  证明：线程何时睡眠、谁唤醒、唤醒后是否运行
  边界：调度事件本身不说明驱动等待条件，需要和 tracepoint/源码合读
```

方法论上，先用 `strace` 找“慢的 syscall”，再用 ftrace/kprobe 找“它进入了哪条内核路径”，再用 tracepoint 找“buffer 状态机卡在哪里”，最后用 perf/sched 判断“是 CPU 热点还是等待/调度”。不要用单一工具解释所有问题。

## 9. 推荐学习路线

按照下面顺序练习，不要一开始就陷入 function_graph 的细节：

1. `v4l2-ctl + strace`

   先掌握用户态 ioctl 生命周期，尤其是 `REQBUFS/QBUF/STREAMON/DQBUF`。

2. `ftrace function`

   把用户态 ioctl 映射到 `v4l2_ioctl -> video_ioctl2 -> video_usercopy -> __video_do_ioctl`。

3. `kprobe + stacktrace`

   精准抓 `vb2_core_qbuf`、`vb2_core_dqbuf` 等关键函数，确认 V4L2 core 到 VB2 的调用栈。

4. `V4L2/VB2 tracepoint`

   从函数调用切换到 buffer 业务时序，观察 `qbuf -> buf_queue -> buf_done -> dqbuf`。

5. `perf record/report`

   学会判断热点到底在 camera 路径、文件 I/O、内存拷贝，还是用户态处理。

6. `sched_* tracepoint / trace-cmd`

   分析线程何时睡眠、何时被唤醒、是否被其他线程抢占。

7. `perf script + FlameGraph`

   把调用栈转换成时间占比视角，适合更复杂的真实系统。

## 10. 最终能力

完成本文实验后，你应该能做到：

- 看到一个 V4L2 抓帧程序，就能画出 `userspace -> ioctl -> V4L2 core -> VB2 -> driver` 的路径。
- 通过 `strace` 判断是不是 `DQBUF` 慢。
- 通过 tracepoint 判断 buffer 是否按预期入队、完成、出队。
- 通过 kprobe 找到 V4L2/VB2 的内核调用栈。
- 通过 perf 判断热点是否来自文件 I/O、内存拷贝、ioctl 处理或其他路径。
- 通过 sched 事件判断用户线程是否被及时唤醒。

这套方法迁移到真实 camera 驱动时，`vivid-000-vid-c` 会换成真实驱动的中断、workqueue 或 kthread；`/dev/video0` 后面也会多出 sensor、CSI、ISP、DMA 等 media entities。但分析方法是同一套。

### 11.9 总结答案

第 9 章推荐路线在当前虚拟机中的复核结论如下：

1. `v4l2-ctl + strace`：正确复现 ioctl 生命周期；`DQBUF` 后续等待约 `0.194s`，符合 5 fps。
2. `ftrace function`：正确映射到 `v4l2_ioctl -> video_ioctl2 -> video_usercopy -> __video_do_ioctl`。
3. `kprobe + stacktrace`：正确证明 `v4l_qbuf -> vb2_ioctl_qbuf -> vb2_core_qbuf`。
4. `V4L2/VB2 tracepoint`：正确观察到 `qbuf -> buf_queue -> buf_done -> dqbuf`。
5. `perf record/report`：正确区分写 raw 文件造成的文件 I/O 热点和不写 raw 时的 V4L2/VB2 ioctl 路径。
6. `sched_* tracepoint / trace-cmd`：正确观察到 `vivid-000-vid-c` 唤醒 `v4l2-ctl`；本次没有明显调度延迟。
7. `perf script + FlameGraph`：`perf script` 可导出栈文本；完整火焰图需要使用完整脚本文件生成。

所以，v2 中关于虚拟机终端输入输出的核心结论可以复现。需要保留的限制是：`trace-cmd report` 在当前 rootfs 中有 plugin/字段解析警告；perf 在 QEMU 中会自动降低采样频率，百分比只适合定性分析。
